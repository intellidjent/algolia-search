Terra release

WorkFusion Terra is a cornerstone release with a number of significant transformations from both the technology and UX perspectives.

One of the biggest fundamental architectural changes in Terra 10.0 is the new bot orchestration platform designed to address current scalability limitations, as well as to reduce complexity and improve fault-tolerance.

Another major improvement is replacing 3 databases (MySQL, PostgreSQL, MongoDB) with a single unified database — MS SQL. This fundamental change allowed us to decrease maintenance complexity and enable consistent data backup, as well as to address security and high availability issues.

Bot Orchestration Platform

In pre-Terra SPA versions, we had the following architecture: Control Tower came as a package with a database, services and infrastructure. But what if we had to handle more tasks and one Control Tower was not enough? We had to add another Control Tower with all its infrastructure, and then another... On top of that, all this needed custom orchestration for any specific use case. This could not last any longer. Such architecture was obviously too complex for large enterprise clients.

In Terra 10.0 the whole architecture has undergone a major re-thinking.

First off, we drew an explicit line between applications, such as Control Tower, and the execution platform. The platform only exposes its API that triggers cluster execution, orchestration, monitoring, and analytics.

Second, the platform dynamically scales the number of workers depending on the task load. This results in lean utilization of resources, which can be used for other tasks.

Last but not least, workers performing tasks are now isolated. This allows to avoid situations when one failed task could lead to unexpected behavior or Control Tower being stuck.

Introducing the new state-of-the-art bot orchestration platform for a horizontally scalable, predictable, and fault-tolerant system. The platform moves the orchestration of Control Tower and AutoML to a separate worker-based service architecture to remove fundamental limitations to the size, complexity and stability of execution.

WorkSpace / TOD

In Terra 10.0 we re-imagined the approach for document labeling when teaching a bot. This approach is naturally familiar to anyone working with physical documents or digital scans, to put it shortly — with original documents.

With this release you no longer need to look for data in an OCRed document version, but rather deal with the original document, just like you'd normally do in a real-life situation.

TOD improves the overall user experience, as well as allows to double the speed of document labeling and cuts the time needed to teach a bot.

Monitoring

We replaced the old Platform Monitor with an open standard solution — ELK stack. In Terra v10.0, ELK does three major things:

  1. Logs aggregation and visualization.
  2. Metrics collection and dashboards with environment status.
  3. Analytics metric collection.

Global logs, environment health metrics, and analytics are collected via Logstash and then conveniently visualized and managed in a single place via Elasticsearch and Kibana.

Deprecations

Since v10.0 is a fresh start release we thought it's high time that we get rid of some legacy items and components that are no longer in focus, or not supported altogether. These items include WorkSpace Sandbox, all Smartcrowd integrations, WFBI service, and a number of minor leftovers scattered around the system.

Here's the list of the most noticeable ones:

  • Platform Monitor is no longer available being replaced by ELK and Bot Manager.
  • Sandbox environment is gone since most of our enterprise customers use a 3-stage environment stack: Dev, UAT, and Production.
